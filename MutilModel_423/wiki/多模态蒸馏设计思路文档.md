# 多模态蒸馏设计思路文档

## 摘要

当前多模态检测蒸馏的核心目标，不是让学生网络机械复刻教师的全部输出，而是把教师在目标存在性判断和边界回归上的有效经验，稳定地迁移到更轻量的学生模型中。基于这一判断，我们将 `output` 蒸馏设计成一套与检测家族紧密绑定的方法，而没有继续沿用统一的、与结构无关的通用输出对齐方案。对于 `YOLOMM`，蒸馏围绕密集预测中的前景位置展开；对于 `RTDETRMM`，蒸馏围绕高质量查询展开。二者共享同一训练范式，即教师固定、学生学习、蒸馏损失与检测损失联合优化，但在监督对象和监督组织方式上分别处理。

## 1. 设计背景

多模态检测模型的蒸馏与单纯分类蒸馏并不相同。教师的输出不仅包含类别判断，还包含空间定位，而且这些信息分布在完全不同的检测结构中。`YOLOMM` 的输出是密集的逐位置预测，天然包含大量低质量背景响应；`RTDETRMM` 的输出则是稀疏的查询集合，真正有效的语义主要集中在少数高质量 query 上。如果仍然使用"对全部输出做统一对齐"的办法，学生会在大量无效监督上消耗容量，蒸馏信号也会被背景噪声稀释。

因此，我们当前的 `output` 蒸馏并不追求形式上的统一，而是先承认不同检测家族的输出结构差异，再为每一类模型单独定义监督对象。换句话说，蒸馏的重点不在"让学生看到教师所有输出"，而在"让学生只学习教师真正有信息量的那部分输出"。

## 2. 总体方法

当前方案采用固定教师的蒸馏范式。教师模型在训练过程中只负责提供参考输出，不参与参数更新；学生模型仍然按原有检测任务完成监督学习，同时额外接收来自教师的输出级约束。这样做的好处是训练目标清晰，教师语义稳定，不会因为共同训练而引入额外的不确定性。

在具体实现上，`output` 蒸馏被拆成两个相互配合的部分。第一部分负责筛选"值得蒸馏"的位置或查询，用来控制监督范围，避免把蒸馏预算浪费在明显无效的背景上。第二部分负责对筛选后的有效预测施加监督，并且把分类与定位分开处理。这样的拆分并不复杂，但很适合检测任务，因为分类误差和定位误差本来就具有不同的统计性质，用一条统一损失硬绑在一起，往往得不到稳定的迁移效果。

此外，我们在训练初期对蒸馏项采用逐步升温的引入方式。这样做的目的很直接：学生在早期首先要站稳自己的基本检测训练过程，如果过早被教师约束得过紧，反而容易破坏原始检测损失的收敛节奏。把蒸馏权重缓慢拉起，本质上是在平衡"先学会检测"与"再学像教师"这两件事。

## 3. YOLOMM 的 output 蒸馏

`YOLOMM` 的输出蒸馏采用前景引导的解耦设计。它的起点不是把整张特征图上的所有位置都拿来对齐，而是先让教师模型给出一个"哪些位置真正值得关注"的判断。只有教师认为具有较高目标置信度的位置，才会进入蒸馏计算。这一步非常关键，因为密集检测头天然带有大比例背景位置，如果不先做筛选，蒸馏信号会被背景主导，学生学到的往往是"如何保持沉默"，而不是"如何做出正确检测"。

在得到前景区域之后，分类与定位被分别处理。分类部分的目标，是让学生在这些前景位置上接近教师的类别判断；定位部分的目标，则是让学生在这些位置上接近教师的边界框表达。这样的设计强调的是"在同一批有效位置上分别继承教师的语义判断和空间回归能力"，而不是让学生去拟合一整块未经筛选的原始输出张量。

从方法气质上看，这一设计更接近面向检测任务的知识迁移，而不是分类蒸馏的直接移植。它保留了教师在前景判别上的选择性，也保留了定位监督在几何上的独立性，因此更适合 `YOLOMM` 这类密集检测结构。

## 4. RTDETRMM 的 output 蒸馏

`RTDETRMM` 的设计思路与 `YOLOMM` 一致，都是先做有效监督筛选，再做解耦蒸馏，但具体对象换成了查询。由于 `RTDETRMM` 的预测单元不是密集位置，而是一组 learned queries，蒸馏时最自然的做法不是对全部 query 一视同仁，而是只保留教师真正有把握的那部分查询。这样做的直接收益，是把蒸馏重点集中在对检测结果真正有贡献的 query 上，而不是让学生去模仿大量低质量、低置信的冗余输出。

在监督层次上，当前方案只针对最终 decoder 层的结果构造蒸馏信号，而没有把所有 decoder 层都纳入统一约束。这个取舍比较保守，但很实用。最终层输出最接近推理时真正使用的检测结果，也最适合作为稳定监督源。对于现阶段的多模态蒸馏来说，这样的约束已经能够覆盖最主要的输出语义，没有必要一开始就把蒸馏路径铺得过宽。

与 `YOLOMM` 类似，`RTDETRMM` 同样把分类与框回归分开建模。区别只在于，`YOLOMM` 的监督载体是前景位置，而 `RTDETRMM` 的监督载体是高质量查询。二者本质上遵循同一原则：蒸馏应该围绕"教师最可信的预测单元"展开，而不是围绕"模型结构里理论上存在的全部输出单元"展开。

## 5. 当前方案与常见蒸馏方法的差异

和早期把教师输出整体展平后直接做回归的方案相比，我们当前的方法更强调结构感知。它不再把所有输出看成同一种张量，而是先区分密集预测和查询预测，再决定监督范围和监督方式。这样做的代价是实现上不再完全统一，但换来的好处是蒸馏信号更贴近真实检测行为。

和经典分类蒸馏相比，当前方案也没有把问题简化成单一的类别 soft target 对齐。检测任务中的分类与定位承担的是两类不同职责，把它们拆开处理，更符合任务本身的性质。再往前一步说，我们当前做的也不是在线蒸馏或共同学习。教师在训练中保持固定，只提供稳定参考；真正被优化的仍然只有学生。这使得整个方法更接近"以成熟教师约束学生"的标准知识蒸馏设定，而不是多模型协同进化。

## 6. 当前实验形态

从现有训练入口来看，仓库中的主实验已经将 `output` 蒸馏作为可直接启用的训练模式。现阶段最清晰的一条实验路径，是以较大的双模态教师模型监督较小的双模态学生模型，并在训练过程中只显式开启 `output` 蒸馏。这个设定说明，当前工作重点已经从"是否支持蒸馏"转向"如何把输出蒸馏做得更符合多模态检测结构"。

这也意味着本文档的重点应当落在方法设计本身，而不是配置细节。配置文件只负责声明教师是谁、哪些教师参与输出蒸馏；真正决定蒸馏行为的，是检测家族对应的输出监督设计。

## 7. Feature 蒸馏

### 7.1 设计原则

Feature 蒸馏不再采用简单的全图 MSE，而是采用**前景引导的检测友好设计**。核心思想是：中间特征的蒸馏应该与检测任务的语义对齐，而不是盲目地让学生在所有空间位置上模仿教师。

这一设计选择的根本原因在于：检测任务中，前景区域和背景区域对模型性能的贡献完全不同。如果使用全图 MSE，蒸馏信号会被大面积背景稀释，学生花大量容量去拟合背景特征，却没有充足的梯度去学习前景表征。前景引导的设计通过教师输出生成的软掩码，将蒸馏监督集中在教师认为重要的区域上。

### 7.2 `role` 的运行时输入语义

`role` 字段的设计严格遵循**运行时输入语义**：

- `role` 表示的是"运行时给该教师喂什么输入"，而非教师的历史训练来源。
- 系统不根据教师内部结构或历史训练方式反向推断 teacher 身份。
- 一个权重被声明为 `role: rgb`，系统就在运行时只取 batch 的 RGB 3 通道喂给它。
- `rgb` 教师与 `x` 教师都可以来自单模态或多模态训练权重，只要前向时按声明输入能够成立。

这意味着 `role` 的职责是确定数据流而非模型属性。用户需要确保声明的 `role` 与教师模型的输入通道兼容，系统在不兼容时直接报错，不做任何猜测或降级。

### 7.3 前景软掩码生成

Feature 蒸馏的第一步是从教师输出生成前景软掩码。掩码的生成完全复用教师的检测输出，不引入额外的分割或注意力模块。

**YOLOMM 教师的掩码生成流程**：
1. 从教师 dense output 中解码所有预测，得到 `boxes_xyxy` 和 `cls_scores`
2. 按 `score >= 0.05` + `topk=100` 双约束筛选教师框
3. 对每个特征映射的目标层 `(B, C, H, W)`，将教师框投影到该层 `(H, W)` 网格
4. 每个框的掩码权重为 `score / sqrt(area + 1e-6)`
5. 多框重叠时做加和，最终 `clamp(mask, 0.0, 1.0)`

**RTDETRMM 教师的掩码生成流程**：
1. 从教师最终 decoder 层输出中取 `boxes_xyxy` 和 `scores`
2. 按 `score >= 0.05` + `topk=100` 双约束筛选有效 query
3. 后续步骤与 YOLOMM 一致（投影、加权、裁剪）

**性能约束**：掩码生成使用批量张量运算（广播、网格比较、批量求和），禁止 Python 级别的逐图逐框双重循环。

### 7.4 最终 4 分项损失公式

对每个 teacher-student 特征映射，计算以下 4 项损失的加权组合：

```
L_map = w_map * (
    lambda_fg  * L_fg
  + lambda_bg  * L_bg
  + lambda_cwd * L_cwd
  + lambda_ctx * L_ctx
)
```

**L_fg（前景损失）**：
- 对 teacher 和 student 的 feature map 做空间归一化（减均值除标准差）
- 计算逐点 `SmoothL1Loss(reduction='none')`
- 按 `fg_mask` 做加权平均
- 这是最主要的蒸馏信号，权重 `lambda_fg = 1.0`

**L_bg（背景损失）**：
- 与 L_fg 计算方式相同，但使用 `bg_mask = 1 - fg_mask`
- 权重 `lambda_bg = 0.25`，作为弱约束
- 目的是防止背景区域特征完全漂移，同时不让背景主导梯度

**L_cwd（通道分布损失 -- Channel-Wise Distribution Distillation）**：
- 对每个通道在前景空间维度展平
- 计算 `softmax(logits / tau)` 后的 KL 散度（tau = 1.0）
- 仅在前景区域上计算（通过 mask 筛选有效空间位置）
- 权重 `lambda_cwd = 0.5`
- 帮助学生学习教师的通道激活分布模式

**L_ctx（上下文语义损失）**：
- 对 fg_mask 加权池化出 `(B, C)` 维度的全局语义向量
- 计算 `1 - cosine_similarity(student_vec, teacher_vec)`
- 权重 `lambda_ctx = 0.25`
- 确保前景区域的高层语义表征一致性

**w_map（分层权重）**：按层深度递增，越接近检测语义末端的层权重越大。多条映射的损失取平均后作为最终 feature 蒸馏损失。

### 7.5 为什么不采用纯 MSE / 纯全图 CWD

纯 MSE 存在三个根本性问题：

1. **前景背景不分**：检测任务中前景和背景的重要性完全不同。MSE 一视同仁会导致蒸馏信号被大面积背景稀释，学生在背景上消耗大量容量却无法有效学习前景表征。

2. **缺乏通道感知**：MSE 只关注逐点差异，完全忽略了通道间的分布关系。实际上，检测模型的中间特征在不同通道上编码了不同的语义模式（边缘、纹理、语义类别等），纯逐点回归无法捕捉这些跨通道的统计规律。

3. **缺乏语义约束**：MSE 是低层度量，关注的是逐像素数值差异，无法保证高层语义的对齐。即使两张 feature map 的 MSE 很小，它们编码的语义含义仍可能存在偏移。

纯全图 CWD 虽然解决了通道感知的问题，但仍然在全图范围上计算，依然面临前景背景不分的困境。

当前方案通过 4 项损失的组合，覆盖了三个维度的对齐需求：
- **空间维度**：L_fg + L_bg 通过掩码区分前景和背景，分别施加不同强度的监督
- **通道维度**：L_cwd 显式对齐通道激活分布
- **语义维度**：L_ctx 通过全局池化后的 cosine 距离保证语义一致

### 7.6 层映射的灵活配置

Feature 蒸馏支持三种配置形式：

1. **原子映射**：单层对单层，适合精确控制
2. **显式区间映射**：多层对多层，适合批量配置
3. **自动展开映射**：教师区间对学生输入侧自动展开，适合多模态架构

所有配置形式在运行时编译阶段统一展开为原子映射列表。系统对展开后的总原子映射数量执行检查（超过 20 条警告，超过 50 条报错），防止映射数量过多导致训练速度显著下降。

### 7.7 Feature 蒸馏的职责

在多模态场景中，Feature 蒸馏承担两个职责：

1. **表征对齐**：帮助学生更早形成与教师相似的中间表征，为后续层提供更好的特征基础
2. **跨模态引导**：在多模态架构中，Feature 蒸馏可以引导学生学习教师的模态融合模式

Feature 蒸馏与 Output 蒸馏是互补的：Output 蒸馏关注最终检测结果，Feature 蒸馏关注中间表征。两者结合使用（`mode='both'`）可以获得更全面的知识迁移。

## 8. 小结

总的来看，当前多模态蒸馏系统的设计思路可以概括为：**让学生只在教师最可信的预测单元上学习教师最有价值的行为，无论是最终输出还是中间特征。**

对于 Output 蒸馏：`YOLOMM` 围绕前景位置做 cls/loc 解耦监督，`RTDETRMM` 围绕高质量 query 做 cls/box 解耦监督。

对于 Feature 蒸馏：两个家族共享同一套前景引导的 4 分项损失公式（L_fg + L_bg + L_cwd + L_ctx），只在上游的教师预测解码和有效框选择上分家。这种设计既保留了检测任务的结构感知，又通过共享损失计算保证了实现的一致性。

结合固定教师范式和渐进式权重引入，整个蒸馏系统在方法上保持了清晰和稳定，同时通过灵活的层映射配置和运行时编译机制，为用户提供了足够的配置弹性。
